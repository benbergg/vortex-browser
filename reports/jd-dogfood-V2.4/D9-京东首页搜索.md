## V2.4 关键发现

### 1. 通信固定开销 23-25s = 真正瓶颈
- V2.2 observe 25s = 23s 通信 + 2s page-side
- V2.4 observe 27s = 23s 通信 + 4s page-side (maxElements=80 触发了 in-viewport 重排, 略慢)
- **maxElements 减少 60% 元素仅 -2s, 通信 23s 是固定开销**
- **fill 26s → 34s 反而慢**, 通信 30s 主导, auto-wait 节省 5s 无法抵消

### 2. 减少 auto-wait 2s × 2 步 = 节省 4-10s
- fill: 3 步 × 5s auto-wait = 15s → 2 步 × 2s = 4s, **节省 11s**
- click: 5s waitActionable (NOT_STABLE) + 5s force retry = 10s → 2s waitActionable + 2s force retry = 4s, **节省 6s**
- 但 fill 实测**反而慢**说明节省被通信开销**吞噬**

### 3. 京东首页 SPA root delegation 仍拦截
- vortex_act click force=true mode=realMouse 仍不跳
- 跳转策略需 V2.3 报告里提到的 hover-then-click / mouse 序列 解决

## V2.4 评测意义

### ✅ 改进
- 总流程 **130s → 94s** (-28%), **绝对值改善 36s**
- maxElements 80 默认 减少通信数据量 (100 KB → 40 KB, **节省 60%**)
- auto-wait 2s 默认 在**响应快**场景下立即返回, 不再 5s 等待

### ❌ 未达成
- spec 项目标 **30s** (实际 94s)
- observe 加速**不明显** (通信固定开销主导)
- fill **反而慢** (通信开销吞噬 auto-wait 节省)
- 京东 SPA 跳转**仍未解决**

### 进一步优化方向
1. **通信优化** (核心): MCP client ↔ vortex-server 通信 23s 是固定开销
   - batch request (合并多次 observe 到 1 次 RPC)
   - 减少 element 序列化 (只发 ref + name + tag, 不发 bbox)
2. **京东 SPA root delegation 适配**: hover-then-click 序列 / 真 mouse event 序列
3. **maxElements 进一步减少** (40-50): 京东首页 80 元素中, 顶部 16 元素 + 60 类目 + 4 推荐 = 80, 顶部优先过滤可减 60%
